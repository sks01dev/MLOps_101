# Day 14: Data Pipelines and Reproducibility with DVC

### The Big Idea: Why Use DVC Pipelines?

In a typical Machine Learning workflow, your data moves through multiple stages in a specific order: raw data $\rightarrow$ cleaned data $\rightarrow$ data splits (train/test) $\rightarrow$ trained model.

#### Without Pipelines: The Problem

If you run your code scripts manually (e.g., typing `python process_data.py` then `python split_data.py`), the system is completely blind to how files connect:

1. **No Dependency Tracking:** If you update your raw data file, your other scripts do not know about it. You have to remember to rerun every single script in the exact right order by hand.
2. **Wasted Compute Time:** If nothing has changed, running the code again forces your computer to recalculate everything from scratch, wasting hours of processing time on large datasets.

#### With Pipelines: The Solution

By writing a **`dvc.yaml`** file, you build a **Directed Acyclic Graph (DAG)** pipeline. This file maps out the exact parent-child relationships of your scripts.

When you type **`dvc repro`**, DVC automatically determines which scripts need to run based on what changed. If a file has not changed, DVC skips running that script entirely and pulls the cached output instantly.

#### The Corrected Configuration Layout (`/root/code/fraud-detection/dvc.yaml`)

```yaml

  process_data:
    cmd: python src/data/process_data.py
    deps:
      - data/raw/transactions.csv
      - src/data/process_data.py
    outs:
      - data/processed/clean_transactions.csv

  split_data:
    cmd: python src/data/split_data.py
    deps:
      - data/processed/clean_transactions.csv
      - src/data/split_data.py
    outs:
      - data/processed/train.csv
      - data/processed/test.csv

```

### The Anatomy of a DVC Stage (The 80/20 Rule)

Every block in a `dvc.yaml` file relies on three foundational properties to track changes:

* **`cmd` (The Execution Trigger):** The exact terminal instruction used to run the code step.
* **`deps` (The Watchlist Inputs):** The file paths DVC closely monitors. If *any* file on this list is modified (whether it's the raw data rows or the script logic itself), DVC flags this stage as "stale."
* **`outs` (The End Products):** The files generated by the execution block. DVC automatically stops Git from tracking these heavy outputs and takes over managing their storage snapshots seamlessly.
